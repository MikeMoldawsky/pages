Delete project — CLI only

Projects are managed via CLI. No TUI project management in MVP.

No agents in project
$ beefy project rm research
  Remove project 'research'? Files stay on beast. [y/N] y
  ✓ Project removed

Files stay on remote machine. Only the project reference is removed from beefy.

Agents running in project
$ beefy project rm myproject
  ⚠ 2 agents are running in 'myproject'.
    They will become standalone sessions.
  Remove project? [y/N] y
  ✓ Project removed. 2 agents are now standalone.

Agents keep running but project column → "—". They can be promoted again later.

Force delete
$ beefy project rm myproject -f
  ✓ Project removed. 2 agents are now standalone.

CLI Commands — MVP

Complete reference. Every TUI action has a CLI equivalent.
TUI = full-screen interactive   Console = scrolling output   SSH = replaces process with SSH   ● interactive = has prompts/pickers   ⚡ scriptable = can skip all prompts with flags

Name uniqueness & generation

All entity names must be unique within their scope. The client sends a name hint, the backend resolves uniqueness.

EntityUnique withinAuto-generated fromOverride flagCollision behavior
Agent Per machine Project name or "session" -n <name> Hint: daemon appends -2, -3.
Override (-n): error if taken.
Project Per machine Repo name (clone) or agent name (save) -n <name> on project save Hint: daemon appends -2, -3.
Override (-n): error if taken.
Machine Global (local config) Hostname from user@host --name <name> on machine new Hint: client appends -2, -3.
Override (--name): error if taken.
How it works: Client always sends a name_hint, never a final name. The backend (daemon for agents/projects, local config for machines) resolves collisions and returns the actual name.
User sees: "Launching myproject-2 on beast..." — the resolved name appears after the backend responds.
Override: When user passes -n or --name, the exact name is sent instead of a hint. Backend errors if taken — no auto-suffix.
Rename collisions: beefy rename old new errors if new is taken. Same for project names.

Notification logic

Trigger states: waiting_input, waiting_permission only (v0).
↓
Delay: Agent enters trigger state → daemon starts timer for notify_delay_secs.
↓
Suppress if: Agent state changes during delay, OR user acts from TUI/CLI during delay.
↓
Fire if: Timer expires AND agent still waiting AND no client activity since state change.
↓
Dedup: Once fired for agent+state, don't re-fire until state changes and returns.
Reply routing: User replies to a specific notification (Telegram reply-to). Daemon maps message_id → agent_id.
Text reply → SendReply RPC. Approve button (callback query) → ApprovePermission RPC. Deny button → DenyPermission RPC.
Stale reply: If agent state changed, bot sends ⚠ instead of forwarding.
Not a reply: Free text without reply-to gets a hint message.
Security: Only paired chat_id receives responses. Unknown users silently ignored.

Daemon config

Stored on each machine at ~/.beefy/config.json. Updated via UpdateConfig gRPC with field mask.

// ~/.beefy/config.json (on remote machine)
{
  "telegram_bot_token": "7891234567:AAH_xxxx...",
  "telegram_chat_id":   "412587349",
  "notify_delay_secs":  120,
  "notifications_paused": false
}
telegram_bot_token — Bot API token from @BotFather. Provider-specific.
telegram_chat_id — Paired Telegram chat ID. Set during setup. Provider-specific.
notify_delay_secs — Seconds before notification fires. Default: 120 (2m). Global — applies to any future provider.
notifications_paused — Mute without removing config. Default: false. Global.
